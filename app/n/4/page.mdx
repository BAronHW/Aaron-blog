# Reactive Programming

While working at Quantexa I have been exposed to a programming paradigm called reactive programming.
Reactive programming allows you to propagate changes through data streams rather than processing data via a pull-based approach.
Instead of explicitly asking for data, your system reacts as new data is pushed through it and different observers receive updates.

Here is an example of the differences between how you would approach the same problem in an imperative style versus a reactive style.
Consider a situation where we want to fetch data when a button is clicked and cancel any previous in-flight request.

## Imperative
```typescript
let currentRequest: AbortController | null = null;

button.addEventListener('click', async () => {
  if (currentRequest) {
    currentRequest.abort();
  }
  currentRequest = new AbortController();

  try {
    const res = await fetch('/api/data', {
      signal: currentRequest.signal,
    });
    render(await res.json());
  } catch (e) {
    if (e.name !== 'AbortError') throw e;
  } finally {
    currentRequest = null;
  }
});
```

## Reactive
```javascript
import { fromEvent } from 'rxjs';
import { switchMap } from 'rxjs/operators';
import { fromFetch } from 'rxjs/fetch';

fromEvent(button, 'click').pipe(
  switchMap(() => fromFetch('/api/data')),
).subscribe(res => render(res));
```

## Explanation

As you can see, the approach taken to solve the same problem is quite different. In the imperative approach we have to manually manage the cancellation of the in-flight request storing a reference to the `AbortController`, checking if one already exists, aborting it, then creating a new one, all while handling errors and cleanup in a `try/catch/finally` block. The state management and bookkeeping obscure what the code is actually trying to do.

In the reactive approach, `switchMap` handles all of this automatically. When a new click event arrives, it cancels the previous observable and starts a new one. The intent of the code is expressed directly: *map each click to a fetch, and cancel any previous fetch in progress*. There is no manual state to manage.

This is the core appeal of reactive programming it lets you describe **what** should happen in response to events, rather than **how** to manage the state required to make it happen.
